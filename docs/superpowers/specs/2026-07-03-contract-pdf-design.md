# Договор поставки + спецификация PDF из данных сделки

**Дата:** 2026-07-03
**Статус:** дизайн утверждён устно, реализация после плана
**Зачем:** закрыть «уйдём в WhatsApp» на шаге после принятия КП — платформа отдаёт готовый к подписи документ. Конкуренты (Supl, Promchat) этого не делают; ЭТП делают через ЭЦП-бюрократию.

## Решение (утверждено)

Серверная генерация PDF через существующую инфраструктуру `export-pdf.js` (pdfkit, JetBrains Mono, рамка + title-block). Никакого хранения документа на платформе: скачал → подписали офлайн (сканы по почте). Загрузка подписанного скана — возможная фаза 2, не сейчас.

Отклонённые альтернативы:
- HTML print-to-PDF — теряет фирменный стиль, зависит от браузера;
- DOCX — новая зависимость, вёрстка с нуля; кандидат на фазу 2, если попросят редактируемый файл.

## 1. БД

В `companies` добавить 6 колонок (обычные `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` в `db.js`, TEXT NOT NULL DEFAULT ''):

| Колонка | Смысл |
|---|---|
| `kpp` | КПП |
| `legal_address` | Юридический адрес |
| `bank_name` | Наименование банка |
| `bank_account` | Расчётный счёт (р/с) |
| `bank_bik` | БИК |
| `bank_corr` | Корреспондентский счёт (к/с) |

ИНН, ОГРН, директор уже есть в `companies` — используются как есть.

## 2. Настройки компании (settings.html)

Новый блок «Реквизиты для договоров»: 6 полей выше + рядом read-only показ ИНН/ОГРН/директора (редактируются там, где редактировались раньше). Сохранение — через существующий route обновления профиля компании (расширить список принимаемых полей). Валидация мягкая: без блокировки сохранения; БИК/р/с/к/с — только цифры, длины 9/20/20, при несовпадении подсветка-предупреждение, не ошибка.

## 3. Endpoint

`GET /api/proposals/:id/contract.pdf?payment=<prepay100|split5050|postpay>`

- Auth: `requireAuth`; доступ только участникам сделки — компания-заказчик заявки или компания-поставщик КП. Остальным 403.
- Условие: КП в статусе `Выигран` (принятое). Иначе 400.
- `payment` невалидный/отсутствует → дефолт `split5050`.
- Реквизиты обеих сторон читаются из `companies` по названию компании (как в остальном коде). Пустое поле → в PDF строка-прочерк `___________________`.
- Реализация — новая build-функция в `export-pdf.js` (`buildContractPdf`), route в `routes/proposals.js` (там уже есть deps и доступ к proposal+order).

## 4. Содержимое PDF

**Договор поставки № ТЗ-2026-<proposalId>**, дата = день генерации, город = город заказчика (или прочерк).

Разделы (~3 страницы):
1. Предмет договора — поставка продукции по Спецификации (Приложение № 1, неотъемлемая часть).
2. Цена и порядок оплаты — сумма из КП прописью и цифрами; вариант оплаты из `payment`:
   - `prepay100` — 100% предоплата в течение 5 банковских дней с даты подписания;
   - `split5050` — 50% аванс в течение 5 банковских дней, 50% в течение 5 банковских дней после приёмки;
   - `postpay` — 100% в течение 10 банковских дней после приёмки.
3. Сроки и условия поставки — срок из КП (дни), отсчёт по варианту оплаты (от аванса либо от подписания).
4. Качество и приёмка — соответствие спецификации/чертежу, приёмка по количеству и качеству, срок заявления претензий.
5. Ответственность сторон — неустойка 0,1%/день, максимум 10% (типовая практика).
6. Разрешение споров — переговоры → претензия (30 дней) → арбитражный суд по месту нахождения истца.
7. Прочие условия — договор в 2 экз., обмен сканами до обмена оригиналами.
8. Реквизиты и подписи сторон — две колонки (Покупатель / Поставщик): наименование, ИНН, КПП, ОГРН, юр. адрес, банк, р/с, к/с, БИК, директор, поле подписи + М.П.

**Приложение № 1 «Спецификация»** (отдельная страница): таблица — наименование (из заявки), категория, количество, цена за единицу (вычисляется, если qty>0), сумма (цена КП), срок поставки, чертёж (имя файла из заявки или «—»). Под таблицей — краткое описание/ТЗ из заявки (обрезка до разумного объёма). Подписи сторон снизу.

Юридический текст — типовой по ГК РФ (§ 3 гл. 30). **До релиза текст проверяет юрист пользователя** — платформа даёт болванку, не юр. заключение.

## 5. UI на странице сделки (deals.html)

Кнопка «Договор + спецификация (PDF)» на карточке/странице сделки, видна обеим сторонам, когда КП принято. Клик → мини-модалка выбора оплаты (3 радио, дефолт 50/50) → кнопка «Скачать» → открывает endpoint с выбранным `payment`. Стиль — существующие чертёжные кнопки (chamfered).

## 6. Ошибки

- Нет реквизитов — НЕ блокировать: генерировать с прочерками, но в модалке показать предупреждение «Реквизиты вашей компании не заполнены — [заполнить в настройках]», если свои поля пустые.
- pdfkit ошибка — существующий обработчик `pipePdf` (500 до headersSent).
- `drawTitleBlocks` уже решает проблему пустых страниц (`margins.bottom=0`).

## 7. Тестирование

- Smoke: расширить `scripts/pdf-smoke.js` вызовом `buildContractPdf` с фикстурой (полные реквизиты / пустые реквизиты / 3 варианта оплаты) — файл генерится, не падает, страниц ≥ 3.
- Route-тест руками: 403 для третьей компании, 400 для непринятого КП.
- Прод-проверка после деплоя: скачать договор по реальной сделке, глазами проверить кириллицу и рамку.

## Вне скоупа (не делать сейчас)

- Хранение/загрузка подписанных сканов.
- DOCX-версия.
- ЭЦП, эскроу.
- Автоподстановка реквизитов по ИНН из внешних API (DaData и т.п.).
